# 1. 索引

索引就是数据的目录，帮助存储引擎快速获取数据的一种数据结构

索引和**数据**都位于**存储引擎**中

#### 常见的两个“存储引擎”

**A. InnoDB (现在的默认老大)**

- **性格**：严谨、安全、细致。
- **核心能力**：
  1. **支持事务 (Transaction)**：如果借书手续办了一半停电了，它能保证回滚，不会出现“书没了但没借书记录”的烂账。
  2. **行级锁 (Row Level Lock)**：你要借《哈利波特》，它只锁这一本书。别人可以同时借旁边的《指环王》。**并发能力强**。
  3. **崩溃恢复 (Crash Safe)**：依靠 Redo Log，断电重启后能把数据救回来。
- **适用场景**：绝大多数现代应用（转账、电商、社交网络）。

**B. MyISAM (以前的老大，已退休)**

- **性格**：简单、粗暴、跑得快（曾经）。
- **核心能力**：
  1. **不支持事务**：出错了没法回滚。
  2. **表级锁 (Table Level Lock)**：你要借《哈利波特》，它把**整个图书馆**封锁了，别人不能动任何书。**并发能力极差**。
  3. **只存数据和索引**，没有复杂的 Crash Safe 机制。
- **适用场景**：极少数只读不写的报表系统（现在基本都不用了，InnoDB 在各方面都超越了它）







# 2. 索引的分类

### 1. 按数据结构分类 (底层算法)

这是索引的**骨架**，决定了索引长什么样，以及它擅长干什么。

- **B+ 树索引 (B+ Tree Index) —— 【默认王者】**
  - **原理**：一种多路平衡查找树。叶子节点存数据，非叶子节点存索引。所有叶子节点用链表连起来。
  - **特点**：
    1. **矮胖**：树高一般只有 2-4 层，磁盘 IO 少。
    2. **有序**：支持范围查询（如 `> 18` 岁）。
  - **场景**：MySQL InnoDB 的默认索引结构，99% 的场景都用它。
  - **比喻**：图书馆的标准分类目录，先找大类，再找小类，最后找到书。
- **哈希索引 (Hash Index)**
  - **原理**：利用哈希表（Key-Value），通过计算 Hash Code 直接定位数据。
  - **特点**：
    1. **极快**：查单条数据（`=`, `IN`）只要 O(1) 的时间。
    2. **缺陷**：**不支持范围查询**（`>`, `<`），因为 Hash 之后原本有序的数字变得无序了。也不支持排序。
  - **场景**：Memory 引擎默认使用。InnoDB 有个“自适应哈希索引”，是自动优化的，不需要你管。
  - **比喻**：指纹解锁。一按手指（Key）直接开门（Value），但你不能说“把指纹纹路数大于 50% 的人都找出来”。
- **全文索引 (Full-text Index)**
  - **原理**：倒排索引 (Inverted Index)。记录“哪个关键词出现在哪篇文章里”。
  - **场景**：搜索引擎场景，如 `select * from t where content like '%Java%'`。
  - **比喻**：书后的“关键词索引表”。查“Java”这个词在第几页出现过。

### 2. 按物理存储分类 (数据在哪)

这是 InnoDB 引擎特有的分类，非常核心。决定了**“你的数据是跟索引住在一起，还是分开住”**。

- **聚簇索引 (Clustered Index) —— 【正文】**
  - **定义**：索引结构的叶子节点存储了**完整的行数据**。
  - **特点**：一张表**只能有一个**聚簇索引（通常是主键）。因为数据只有一份，不能同时放在两个地方。
  - **比喻**：字典的正文。你查到“A”这个字，解释（数据）就在旁边写着。
- **二级索引 / 非聚簇索引 (Secondary Index) —— 【目录】**
  - **定义**：索引结构的叶子节点存储的是**主键 ID**，而不是真实数据。
  - **特点**：一张表可以有多个二级索引（比如给 name 建一个，给 age 建一个）。
  - **代价**：**回表**。如果你查 `name='bao'`，找到 ID 后，通常还需要拿着 ID 回到聚簇索引里去查完整数据，这个过程叫做回表
  - **比喻**：字典前面的“拼音目录”。你查到“bao”，目录只告诉你“去第 100 页找”。

### 3. 按字段特性分类 (逻辑约束)

这是我们在 `CREATE TABLE` 时最关心的逻辑属性。

- **主键索引 (Primary Key)**
  - **特性**：**唯一 + 非空 (NOT NULL)**。
  - **地位**：它是聚簇索引的“宿主”。每张表都应该有一个。
  - **比喻**：身份证号。全中国唯一，且每个人必须有。
- **唯一索引 (Unique Index)**
  - **特性**：**唯一**，但**允许为空 (NULL)**。
  - **作用**：除了加速查询，主要是为了限制数据约束（比如手机号注册，不能重复）。
  - **比喻**：手机号。不能重复，但有的人可能还没办手机号（NULL）。
- **普通索引 (Normal Index)**
  - **特性**：没有任何约束，允许重复，允许 NULL。
  - **作用**：纯粹为了**快**。
  - **比喻**：姓名。可以重复（重名），建立索引只是为了快速找到叫“张三”的人。
- **前缀索引 (Prefix Index)**
  - **特性**：只取字符串的前几个字符建立索引。
  - **场景**：比如存 Email 或很长的 URL。全存下来索引太大了，只存前 6 位通常就能区分大部分数据。
  - **优点**：省空间。
  - **缺点**：不能用于 `ORDER BY` 和覆盖索引。

### 4. 按字段个数分类 (组合策略)

这是优化查询性能的重头戏。

- **单列索引**
  - 即一个索引只包含一个列。
  - 例如：`idx_name(name)`
- **联合/复合索引 (Joint / Composite Index)**
  - 即一个索引包含了多个列。
  - 例如：`idx_name_age(name, age)`。
  - **核心考点**：
    1. **最左前缀原则**：查 `name` 走索引，查 `name + age` 走索引，单查 `age` **不走索引**。
    2. **覆盖索引 (Covering Index)**：如果查询 `select name, age from t where name='Alice'`，数据都在索引树上，**不需要回表**，性能起飞。
  - **比喻**：电话簿。先按【姓】排序，同姓的再按【名】排序。你直接找【名】是找不到的。



### A 联合索引范围查询 (The "Red Light" Rule)

这是面试必考题：**“为什么范围查询会导致索引失效？”**

#### 1. 核心结论

在联合索引 `(a, b, c)` 中：

- **遇到等值查询**（`=`, `IN`）：绿灯，继续匹配下一个字段。
- **遇到范围查询**（`>`, `<`, `BETWEEN`）：**红灯**，当前字段能用索引，但**后面的字段无法使用索引**。

#### 2. 原理拆解（为什么？）

还是用 **(班级, 年龄)** 的例子：

- **查询**：`WHERE 班级 > 1 AND 年龄 = 10`
- **现状**：
  - **班级 > 1**：这个范围跨越了 2 班、3 班、4 班……
  - **年龄**：在 2 班内部是有序的，在 3 班内部也是有序的。但是！**把 2 班和 3 班放在一起看，年龄是忽大忽小的，不再整体有序。**
- **结果**：索引只能帮你快速定位到“2 班”的开头，之后只能一个个往后找，没办法直接跳到 `年龄=10` 的位置。

------

### B 索引下推 (ICP - Index Condition Pushdown)

这是 MySQL 5.6 引入的“补救措施”。

#### 1. 它是干嘛的？

当联合索引因为“范围查询”或者“模糊匹配”导致后续字段失效时，**虽然无法利用索引进行“查找（定位）”，但可以利用索引进行“过滤”。**

#### 2. 场景复现

索引 `(name, age)`。

SQL：`SELECT * FROM user WHERE name LIKE '张%' AND age = 10;`

- **没有 ICP (旧版)**：
  - MySQL 只看 `name` 索引。找到所有姓“张”的，可能有 100 个。
  - 全部**回表** 100 次，把完整数据拿回 Server 层。
  - Server 层再一个个检查 `age` 是不是 10。
  - **代价**：100 次回表（IO 爆炸）。
- **有了 ICP (新版)**：
  - InnoDB 引擎在索引树上看到 `name` 是“张三”时，**顺便看一眼旁边的 `age`**。
  - 如果 `age != 10`，**直接扔掉，不回表**。
  - 只有完全匹配的，才回表。
  - **收益**：可能只需要 1 次回表。

------

### C 索引区分度 (Index Selectivity)

这是设计索引时的**黄金法则**：**“谁排在联合索引的最左边？”**

#### 1. 定义

**区分度 = 不重复的索引值个数 / 表记录总数**

- 值越接近 **1**，区分度越好（如 ID、手机号）。
- 值越接近 **0**，区分度越差（如 性别、状态）。

#### 2. 为什么重要？

想象你在修剪索引树：

- 如果你把 **“性别”** 放在最左边 `(sex, name)`：
  - 你要找“男”，一下捞出来 50% 的数据。这棵树的第一层只有两个叉，过滤效果极差。
- 如果你把 **“名字”** 放在最左边 `(name, sex)`：
  - 你要找“张三”，一下就只剩下几个人了。过滤效果极好。

#### 3. 最佳实践

**尽量把区分度高的列放在联合索引的最左边。**

（但在某些特殊场景下，为了配合 `ORDER BY` 排序或覆盖索引，可能会打破这个规则）。

------

### D 联合索引进行排序 (Order By)

很多时候，我们建索引不仅仅是为了 **“查” (Where)**，还是为了 **“排” (Order By)**。

#### 1. 利用索引排序 (Using index)

因为 B+ 树本身就是有序的，如果你的 SQL 顺序和索引顺序一致，MySQL 就**不需要**额外消耗 CPU 去排序了。

索引 `(a, b, c)`：

- `ORDER BY a, b, c` -> **完美**（利用索引顺序）。
- `WHERE a = 1 ORDER BY b, c` -> **完美**（a 定值后，b 和 c 就是有序的）。
- `ORDER BY a DESC, b DESC` -> **完美**（MySQL 8.0 支持降序索引）。

#### 2. 索引排序失效 (Using filesort)

如果打破了顺序，MySQL 就只能把数据取出来，在内存（Sort Buffer）里自己重新排一遍，这叫 **FileSort**（由于可能涉及磁盘临时文件，效率低）。

- `ORDER BY a, c` -> **失效**（跳过了 b，中间断了）。
- `WHERE a > 1 ORDER BY b` -> **失效**（a 是范围查询，导致后面的 b 无序了）。
- `ORDER BY a ASC, b DESC` -> **失效**（一正一反，除非建索引时指定了方向）。

------

### 总结图谱

| **知识点**   | **核心口诀**               | **关键点**                                              |
| ------------ | -------------------------- | ------------------------------------------------------- |
| **范围查询** | **“最左匹配，遇范围即停”** | `>` `<`  会阻断后续字段的索引查找。                     |
| **索引下推** | **“引擎层先过滤，少回表”** | 虽然查找断了，但能顺便过滤垃圾数据，减少 IO。           |
| **区分度**   | **“谁独特谁靠左”**         | 区分度高（接近1）的列放左边，筛选效率高。               |
| **排序**     | **“顺序一致才免费”**       | `Order By` 的字段顺序要和索引一致，且不能跨过范围查询。 |





# 3. 索引使用场景

### 🟢 绿灯：必须/建议建索引的情况

这些场景下，不建索引就是“耍流氓”，数据库会慢到怀疑人生。

1. **字段有唯一性限制 (Unique)**
   - **理由**：这是业务刚需。比如“用户名”、“手机号”、“身份证号”。业务上要求唯一，就必须建唯一索引。
2. **经常用于 `WHERE` 查询的字段**
   - **理由**：这是索引最基础的用法。你查什么，就给什么建目录。
3. **经常用于 `GROUP BY` 和 `ORDER BY` 的字段**
   - **理由**：
     - 如果不建索引，MySQL 就要把数据全拿出来，在内存（Sort Buffer）里重新排序或分组（Using filesort / Using temporary）。
     - 如果建了索引，B+ 树本身就是有序的，直接按顺序拿就行了（**天然有序**），效率提升巨大。
4. **`UPDATE`、`DELETE` 的 `WHERE` 条件列**
   - **理由**：如果不走索引，这些操作可能会导致**锁全表**（如果你用的是 MyISAM）或者扫描全表，非常危险。
5. **`DISTINCT` 字段**
   - **理由**：和 `GROUP BY` 类似，索引能加速去重。
6. **多表 `JOIN` 的连接字段**
   - **理由**：
     - 例如 `select * from A join B on A.id = B.user_id`。
     - 这个 `B.user_id` 必须建索引！
     - **注意**：连接字段的**数据类型必须一致**，否则索引会失效（隐式类型转换）。
7. **列的区分度（Cardinality）很高**
   - **理由**：区分度越高，索引效果越好。
   - **公式**：`count(distinct col) / count(*)` 越接近 1 越好。
   - **例子**：UUID、手机号、邮箱。

------

### 🔴 红灯：不建议/禁止建索引的情况

有些时候，建了索引不仅没用，反而会拖慢系统。

1. **`WHERE` 条件里用不到的字段**
   - **理由**：不要为了建索引而建索引。没人查它，建它干嘛？浪费空间，还拖慢插入速度。
2. **表的数据量太少 (比如少于 1000 行)**
   - **理由**：
     - 比如一个“配置表”，就几行数据。
     - 索引文件本身也有开销。对于这种微型表，**全表扫描可能比查索引还要快**（因为不需要回表，也不需要读索引页）。
3. **字段区分度太低 (Low Cardinality)**
   - **理由**：
     - 典型例子：**“性别”**（只有男女）、**“状态”**（只有 0/1）。
     - 如果你建了索引，一查“男”，可能一下捞出一半的数据。MySQL 优化器会觉得：*“反正都要读一半的数据，还要回表，不如直接全表扫描算了。”* 于是它会弃用你的索引。
4. **经常更新的字段**
   - **理由**：
     - 每次 `UPDATE` 这个字段，不仅要改数据，还要去 B+ 树里拔掉旧节点、插入新节点，甚至引发**页分裂 (Page Split)**。
     - 如果这个字段读取很少，写入极多，建索引就是负收益。
5. **字段太长 (Text / Long Varchar)**
   - **理由**：
     - 索引文件会变得巨大，不仅占磁盘，还挤占内存（Buffer Pool）。
     - **解决方案**：使用**前缀索引**（只索引前 10-20 个字符）。
6. **已经有联合索引了，不需要重复建**
   - **理由**：
     - 如果你已经有联合索引 `(a, b)`。
     - 就**千万不要**再单独给 `a` 建一个索引了！
     - 根据**最左前缀原则**，`(a, b)` 已经能覆盖 `a` 的查询了。





# 4. 索引优化方法

索引优化其实是一门**“做减法”**的艺术。

它的核心目标只有两个：

1. **让 B+ 树更小、更瘦**（减少磁盘 IO）。
2. **让查询路径更短**（减少回表，减少扫描行数）。

------

### 一、 设计层面的优化（给 B+ 树“瘦身”）

#### 1. 前缀索引 (Prefix Index) —— 专治长字符串

如果你有一个 `address` 字段（VARCHAR 255），里面存的都是很长的地址。

- **笨办法**：`KEY idx_addr (address)`。
  - **缺点**：索引文件巨大，一页存不了几个索引，查询慢。
- **优化办法**：**只索引前 N 个字符**。
  - 通常前 10~20 个字符就足够区分大部分地址了。
  - **SQL**：`ALTER TABLE t ADD KEY idx_addr_prefix (address(10));`
  - **效果**：索引体积可能缩小 10 倍，IO 效率大幅提升。

#### 2. 区分度优先 (Cardinality)

我们在前一个问题里提到了，区分度低的字段别建索引。

- **优化策略**：在建立**联合索引**时，把区分度高的字段放在**最左边**。
- **例子**：`(status, user_id)` vs `(user_id, status)`。
  - `status` 只有 0/1，`user_id` 有几百万个。
  - **绝对选** `(user_id, status)`。因为先定 ID，一下子就锁定了目标；先定状态，还是捞出一半数据。

#### 3. 主键最好是自增

- **原因**：你还记得吗？**所有二级索引（普通索引）的叶子节点里，都存了主键 ID。**
- **后果**：如果你的主键是一个很长的字符串（比如 32 位的 UUID），那么**每一个**二级索引都会变得非常胖，占用大量磁盘空间。
- **建议**：尽量使用 `BIGINT` 自增主键（8 字节），不仅省空间，而且插入时是顺序的，减少页分裂（为了在已满的数据页中间插入新数据，不得不申请新页，并将数据大规模挪动的过程。此过程代价十分昂贵）。

------

### 二、 查询层面的优化（抄近道）

#### 4. 覆盖索引 (Covering Index) —— 拒绝回表

这是最高效的优化手段。

- **场景**：你需要查 `select name, age from t where name = 'Zhang';`
- **优化**：建立联合索引 `(name, age)`。
- **效果**：
  - MySQL 在索引树上找到 'Zhang' 时，发现旁边就躺着 `age`。
  - **直接返回！** 不需要再去翻阅 16KB 的数据页了。
  - **关键词**：在 `Explain` 结果中看到 `Using index`，说明优化成功。

------

### 三、 避坑层面的优化（防止索引失效）

有时候索引建好了，但你写了一句烂 SQL，导致 MySQL **“被迫”** 放弃索引，改为全表扫描。以下是**“索引失效四大杀手”**：

#### 5. 不要在索引列上做计算/函数

这是新手最容易犯的错。

- **错误写法**：

  SQL

  ```
  select * from t where YEAR(create_time) = 2024;
  ```

  - **原因**：B+ 树里存的是具体的 `2024-02-11 12:00:00`，不是 `2024`。你对它用了函数，它就“不认识”原来的树结构了，只能全表扫描，一个个算出来再比对。

- **优化写法**：

  SQL

  ```
  select * from t where create_time between '2024-01-01' and '2024-12-31';
  ```

  - 把计算放到“等号右边”，让索引列保持“干净”。

#### 6. 警惕隐式类型转换 (Implicit Conversion)

这通常发生在 `VARCHAR` 和 `INT` 混用的时候。

- **场景**：`phone` 字段是 `varchar(11)`，且有索引。

- **错误写法**：

  SQL

  ```
  select * from t where phone = 13800001234;  -- 注意这里没加单引号，是数字
  ```

- **后果**：MySQL 会在内部把每一行的 `phone` 转成数字再比较。相当于用了函数 `CAST(phone AS SIGNED)`。**索引直接失效！**

- **优化**：`where phone = '13800001234'` (加上单引号)。

#### 7. 避免以 `%` 开头的模糊查询

- **错误**：`where name like '%三'`。
  - **原因**：B+ 树是按照“首字母”排序的。你连首字母都不给我，我怎么查字典？只能从头翻到尾。
- **优化**：
  - 尽量用 `like '张%'` (索引有效)。
  - 如果非要查 `%三`，考虑用 **全文索引 (Fulltext)** 或者 **Elasticsearch**。

#### 8. 少用 `OR`，多用 `UNION` 或 `IN`

在某些旧版本的 MySQL 中，索引 `(a)` 和 索引 `(b)`。

- `select * from t where a=1 OR b=2;`

- 可能导致索引失效。因为 MySQL 一次查询通常只能用一个索引（除非触发索引合并 Index Merge，但效率也不高）。

- **优化**：

  SQL

  ```
  select * from t where a=1
  UNION ALL
  select * from t where b=2;
  ```

------

### 四、 终极武器：Explain

你要想知道优化有没有生效，不能靠猜。

在你的 SQL 前面加一个 `EXPLAIN`：

SQL

```
EXPLAIN select * from user where id = 1;
```

重点看这三列：

1. **`type`**：访问类型。
   - 看到 `ALL`？**完蛋，全表扫描了**。赶紧优化。
   - 看到 `ref` 或 `range`？**还不错**。
   - 看到 `const` 或 `system`？**完美**。
2. **`key`**：实际用到的索引。
   - 如果是 `NULL`，说明没用上索引。
3. **`Extra`**：
   - 看到 `Using filesort`？**糟糕**，需要额外排序（通常因为 `order by` 没用上索引）。
   - 看到 `Using temporary`？**糟糕**，用到了临时表（通常因为 `group by` 或 `distinct`）。
   - 看到 `Using index`？**真香**，覆盖索引生效了。

### 总结

优化索引就像**“整理书架”**：

1. **书签做短一点**（前缀索引）。
2. **常用的书放手边**（区分度高放左边）。
3. **不要在书名上乱涂乱画**（不在列上做运算）。
4. **直接看目录解决问题，别老翻书**（覆盖索引）。
