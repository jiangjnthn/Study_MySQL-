# 一、InnoDB 是如何存储数据的？

InnoDB 的数据是按**「数据页」为单位**来读写的，InnoDB **数据页的默认大小是 16KB**，意味着数据库每次读写都是以16KB 为单位的，一次最少从磁盘中读取 16K 的内容到内存中，一次最少把内存中的 16K 内容刷新到磁盘中。

你可以把它想象成**一张标准的 A4 纸**，或者**一间教室**。这 16KB 的空间被精准地划分成了 **7 个部分**。

这 7 个部分按照从上到下的物理顺序排列如下：

### 1. File Header (文件头) —— 38 B

**作用：页的“身份证”和“对外联络员”。**

这是页的最头部，主要包含两类关键信息：

- **页号 (Page Number)**：告诉系统“我是第几页”。
- **前后指针 (FIL_PAGE_PREV / FIL_PAGE_NEXT)**：
  - 这是最核心的设计！它存储了**上一页**和**下一页**的页号。
  - **意义**：这使得所有的数据页在逻辑上形成了一个**双向链表**。哪怕物理上这两个页在磁盘里隔得很远，逻辑上它们是紧紧相连的。这也正是 B+ 树叶子节点能进行范围查询的基础。

![image-20260212170020782](C:\Users\lenovo\AppData\Roaming\Typora\typora-user-images\image-20260212170020782.png)

- **校验和 (Checksum)**：用于检查页是否完整，保证了数据的原子性（配合文件尾部使用）。

### 2. Page Header (页头) —— 56 B

**作用：页的“状态仪表盘”。**

`File Header` 关注的是页和页之间的关系，而 `Page Header` 关注的是**本页内部**的状态。

- 它记录了：
  - 本页存了多少条记录？
  - 第一条记录在哪里？
  - 空闲空间在哪里？
  - 页目录（Page Directory）里有多少个槽？
- **意义**：存储引擎操作这一页时，先看这里，就能快速知道这页满没满，还能塞多少数据。

### 3. Infimum + Supremum (最小和最大虚记录) —— 26 B

**作用：页内记录链表的“两个锚点”。**

这是两个**虚拟**的记录，你哪怕建个空表，页里也有这两个东西。

- **Infimum (最小虚记录)**：比页内任何真实记录都小。
- **Supremum (最大虚记录)**：比页内任何真实记录都大。
- **意义**：
  - 页里的真实数据是一个**单向链表**。
  - 为了链表操作方便（始终有头有尾），InnoDB 规定：链表从 Infimum 开始，串联所有真实记录，最后指向 Supremum 结束。它俩就像是链表的**哨兵**。
  - 现在的链表结构永远是这样的：Infimum -> [真实记录 A] -> [真实记录 B] -> ... -> Supremum
- 作用：简化链表操作；锁的依附点
- 总结：最大最小虚记录就是链表的**“固定书立（书挡）”**。不管中间夹了多少本书（数据），这两个书立永远在两头撑着，方便管理，还能用来挂锁。

### 4. User Records (用户记录) —— 大小不固定

**作用：真正存数据的地方。**

- 这里就是我们在前面讲过的 **COMPACT 行格式**（包裹单+包裹货品）存放的地方。
- **生成方式**：一开始这个区域是空的。每当你插入一条数据，就会从 **Free Space** 里切一块下来，变成 **User Records**。
- **排列逻辑**：物理上可能是乱序的（因为删除、插入会导致碎片），但在**逻辑上**通过 `next_record` 指针连成一个按主键排序的单向链表。

### 5. Free Space (空闲空间) —— 大小不固定

**作用：未分配的备用空间。**

- 这就好比那张 A4 纸上还没写字的**空白区域**。
- 当一条新记录插入时，Free Space 减少，User Records 增加。
- 当 Free Space 用完了，说明这一页满了，就需要申请新的页（或者发生页分裂）。

### 6. Page Directory (页目录) —— 大小不固定

**作用：页内的“稀疏索引” (加速器)。**

这是一个极其重要的设计。

**问题**：User Records 是个单向链表。如果一页里有 1000 条数据，你想找第 900 条，难道要从头遍历 900 次吗？那太慢了（O(N)）。

**解决**：

- InnoDB 把记录分组（比如每 4-8 条一组）。
- 取出每组最后一条记录的地址，存到 **Page Directory** 里，这就叫 **“槽” (Slot)**。（**每个槽相当于指针指向了不同组的最后一个记录**）
- **搜索过程**：
  1. 先在 Page Directory 里用 **二分查找 (Binary Search)**，快速锁定目标记录在哪一个“组”里（O(logN)）。
  2. 再去那个组里，线性遍历几条数据找到目标。
- **位置**：注意，它是从页的**最底部**倒着往上长的。

### 7. File Trailer (文件尾部) —— 8 B

**作用：页的“安全卫士” (完整性校验)。**

- **核心痛点**：刷脏页。
  - 内存里改好了一页，正准备写回磁盘。
  - 写了 8KB（刚写了一半），**断电了！**
  - 这时候磁盘里的页是坏的（一半新一半旧）。
- **解决**：
  - `File Trailer` 里存了一个校验和（Checksum）。
  - 写入磁盘时，先算好头部的 Checksum，写完尾部时再写一个 Checksum。
  - 读取时，如果**头部和尾部的 Checksum 对不上**，说明上次写入没写完（Torn Page），这页数据损坏了。

### 总结图示

把这一页想象成一本书：

| **区域**             | **对应书的结构** | **核心作用**                                         |
| -------------------- | ---------------- | ---------------------------------------------------- |
| **File Header**      | **页码与装订线** | 记录“第几页”，并用线把这一页和上一页、下一页连起来。 |
| **Page Header**      | **本页摘要**     | 记录“本页有多少字，还有多少空白”。                   |
| **Infimum/Supremum** | **页眉和页脚**   | 正文的起始线和结束线，正文不能写出这个范围。         |
| **User Records**     | **正文内容**     | 你真正写的文字（数据行）。                           |
| **Free Space**       | **空白处**       | 还没写字的地方。                                     |
| **Page Directory**   | **段落索引**     | 记录每个段落的开头在哪一行，方便你快速跳读。         |
| **File Trailer**     | **防伪水印**     | 核对这一页有没有被撕坏或印刷不完整。                 |











# 二、B+树是如何进行查询的？

- 只有叶子节点(最底层的节点)才存放了数据，非叶子节点（其他上层节）仅用来存放目录项作为索引

- 非叶子节点分为不同层次，通过分层来降低每一层的搜索量

- 所有节点按照索引键大小排序，构成一个双向链表，便于范围查询

我们再看看B+树如何实现快速查找主键为6的记录，以图为例子:

![image-20260213221046071](C:\Users\lenovo\AppData\Roaming\Typora\typora-user-images\image-20260213221046071.png)从根节点开始，通过二分法快速定位到符合页内范围包含查询值的页，因为查询的主键值为6，在[1,7)范围之间，所以到页30中查找更详细的目录项

在非叶子节点(页30)中，继续通过二分法快速定位到符合页内范围包含查询值的页，主键值大于5，所以就到叶子节点(页16)查找记录

接着，在叶子节点(页16)中，通过槽查找记录时，使用二分法快速定位要查询的记录在哪个槽(哪个记录分组)，定位到槽后，再遍历槽内的所有记录，找到主键为6的记录

可以看到，在定位记录所在哪一个页时，也是通过二分法快速定位到包含该记录的页。定位到该页后，又会在该页内进行二分法快速定位记录所在的分组(槽号)，最后在分组内进行遍历查找。











# 三、聚簇索引和二级索引

另外，索引又可以分成聚簇索引和非聚簇索引(二级索引)，它们区别就在于叶子节点存放的是什么数据:

- 聚簇索引的叶子节点存放的是实际数据，所有完整的用户记录都存放在聚簇索引的叶子节点

- 二级索引的叶子节点存放的是主键值，而不是实际数据。

因为表的数据都是存放在聚簇索引的叶子节点里，所以 **InnoDB 存储引擎一定会为表创建一个聚簇索引**，且由于数据在物理上只会保存一份，所以**聚簇索引只能有一个**。

InnoDB在创建聚簇索引时，会根据不同的场景选择不同的列作为索引:

- 如果有主键，默认会使用主键作为聚簇索引的索引键
- 如果没有主键，就选择第一个不包含NULL值的唯一列作为聚簇索引的索引键
- 在上面两个都没有的情况下，InnoDB将自动生成一个隐式自增id列作为聚簇索引的索引键

一张表只能有一个聚簇索引，那为了实现非主键字段的快速搜索，就引出了二级索引（非聚簇索引/辅助索引），它也是利用了B+树的数据结构，但是二级索引的叶子节点存放的是主键值，不是实际数据。

因此，如果某个查询语句使用了二级索引，但是查询的数据不是主键值，这时在二级索引找到主键值后:需要去聚簇索引中获得数据行，这个过程就叫作**「回表」**，也就是说要查两个B+树才能查到数据。不过，当查询的数据是主键值时，因为只在二级索引就能查询到，不用再去聚簇索引查，这个过程就叫作**「索引覆盖」**，也就是只需要查一个B+树就能找到数据。









# 四、总结

**三个层次（物理存储 -> 页内结构 -> 索引架构）** 

------

### 第一层：宏观的物理存储（数据页与链表）

> **原文**：“InnoDB 的数据是按「数据页」为单位来读写的，默认数据页大小为 16 KB。每个数据页之间通过双向链表的形式组织起来，物理上不连续，但是逻辑上连续。”

- **解读**：
  - **最小单位**：你写一行数据，MySQL 不是只写那一行，而是把这行所在的**整整 16KB 的页**加载进内存，改完后再刷回磁盘。这叫“页是 IO 的基本单位”。
  - **双向链表**：还记得数据页结构里的 `File Header` 吗？它里面有两个指针（上一页、下一页）。这就保证了哪怕这些页散落在磁盘的不同位置（物理不连续），MySQL 也能顺着指针找到前一页和后一页（逻辑连续）。
  - **作用**：这个双向链表是执行**全表扫描**和**范围查询**（比如 `id > 100`）的基础。

------

### 第二层：微观的页内结构（单向链表与页目录）

> **原文**：“数据页内包含用户记录，每个记录之间用单向链表的方式组织起来... 设计了一个页目录，页目录存储各个槽（分组），且主键值是有序的，于是可以通过二分查找的方式进行检索从而提高效率。”

- **解读**：
  - **单向链表**：在页内部，每一行数据（User Record）通过 `next_record` 指针连在一起。这就像一个班级的学生手拉手排成一列。
  - **痛点**：如果要找学号是 50 的学生，顺着手拉手找太慢了（O(N)）。
  - **解决方案（页目录）**：这就是我们刚讲的**“分组查人”**。把学生分成小组，选出组长，老师（MySQL）手里拿一张表记录组长的位置（槽 Slot）。
  - **效率**：先用**二分查找**快速找到是哪个组的（O(logN)），再去那个组里遍历（最多遍历 8 次）。这是保证单页查询速度极快（毫秒级）的关键。

------

### 第三层：宏观的索引架构（B+树、聚簇/二级索引、回表）

> **原文**：“为了高效查询记录所在的数据页，InnoDB 采用 b+ 树作为索引... 如果叶子节点存储的是实际数据就是聚簇索引... 如果存储的不是实际数据，而是主键值则是二级索引...”

- **解读**：
  - **B+树的本质**：它其实就是把上面讲的无数个“数据页”，按照层级堆叠起来的一种目录结构。
  - **聚簇索引 (Clustered Index)**：
    - **特点**：叶子节点 = **完整数据**。
    - **地位**：就是表本身。找到索引就找到了数据。
  - **二级索引 (Secondary Index)**：
    - **特点**：叶子节点 = **索引列 + 主键 ID**。
    - **地位**：是辅助目录。它不存完整数据，只存一个“引用”（主键 ID）。

> **原文**：“...如果查询的数据不在二级索引里，就需要先在二级索引找到主键值，需要去聚簇索引中获得数据行，这个过程就叫作「回表」。”

- **解读（核心考点）**：
  - **回表 (Back to Table)**：这是性能杀手。
  - **举例**：有一张表 `(id, name, age)`，主键 `id`，二级索引 `name`。
    - 查 `select id, name from t where name='张三'` -> **不需要回表**（因为 `name` 索引里就有 id 和 name）。这就叫**索引覆盖**。
    - 查 `select age from t where name='张三'` -> **需要回表**（因为 `name` 索引里没有 age，必须拿着 id 回到聚簇索引里去找）。

------

### 总结（一句话串联）

这段文字描述了数据在 MySQL 里的一次完整旅程：

1. 数据乖乖地躺在 **16KB 的页** 里。
2. 在页内部，通过 **页目录** 让你能瞬间找到它。
3. 页和页之间，通过 **双向链表** 连成一片。
4. 为了管理成千上万的页，我们盖了一栋楼叫 **B+ 树**。
5. 想找人（查数据），先查 **二级索引**，如果资料不够，再拿身份证号（主键）去 **聚簇索引**（大本营）里调档案（**回表**）。
