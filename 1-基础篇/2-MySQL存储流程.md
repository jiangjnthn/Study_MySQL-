# 第一阶段：宏观物理存储——数据到底怎么躺在硬盘上？

#### **第一部分：MySQL 的数据存放在哪个文件？**

当你执行 `INSERT INTO student ...` 时，数据最终是落到了磁盘上的某个文件里。这个行为完全由 **存储引擎** 决定。

在 Linux 下，首先去查看你的 MySQL 配置文件 `my.cnf` 中的 `datadir` 参数，通常默认在 `/var/lib/mysql/`。

进入这个目录，你会看到以下几类核心文件：

1. **`db.opt`** (旧版本) / **目录本身**：用来存放数据库的默认字符集等配置。
2. **`.frm` 文件** (MySQL 8.0 之前)：
   - **作用**：存放**表结构定义**（Table Definition）。
   - **细节**：不管你用什么存储引擎（InnoDB 或 MyISAM），都会有这个文件。但在 **MySQL 8.0** 之后，这个文件被移除了，表结构定义被整合到了系统表空间（System Tablespace）里的一组特殊表中。这组特殊的表叫 **数据字典 (Data Dictionary)**。
3. **`.ibd` 文件** (InnoDB Data)：
   - **含义**：本身是一颗B+树。
   - **作用**：存放 **InnoDB 表的数据** 和 **索引**
   - **关键参数 `innodb_file_per_table`**：
     - **ON (推荐/默认)**：**独占表空间**。每个表都有一个独立的 `.ibd` 文件（如 `student.ibd`）。这样做的好处是，你 `DROP TABLE` 时，磁盘空间会直接由操作系统回收。
     - **OFF**：**共享表空间**。所有表的数据都混在一个巨大的 `ibdata1` 文件里。这非常坑，因为即使你删了表，`ibdata1` 也不会变小，磁盘空间很难释放。

------

#### 第二部分：表空间文件（Tablespace）的结构是怎么样的？

表空间由段(segment)、区(extent)、页(page)、行(row)组成，InnoDB存储引擎的逻辑存储结构，大致如下图：

<img src="C:\Users\lenovo\AppData\Roaming\Typora\typora-user-images\image-20260123194406322.png" alt="image-20260123194406322" style="zoom:67%;" />

当我们打开一个 `student.ibd` 文件，它内部绝不是杂乱无章的。它有着极其严格的 **层级管理结构**。

我们可以把它想象成一个**图书馆管理系统**：

**Tablespace (图书馆) -> Segment (书架区) -> Extent (书架) -> Page (书页) -> Row (文字行)**

让我们逐层拆解细节：

**① 页面 (Page) —— 最小 IO 单位**

这是最核心的概念。

- **大小**：默认 **16KB**。
- **意义**：MySQL 和磁盘交互时，**从来不会只读“一行”数据**。哪怕你只查 id=1 的那一行，InnoDB 也会把这行所在的**整整 16KB 的页**加载到内存（Buffer Pool）中。
- 页的类型有很多，常见的有数据页、undo 日志页、溢出页等等。数据表中的行记录是用「数据页」来管理的，表中的记录存储在数据页里

**② 区 (Extent) —— 空间分配单位**

- **大小**：**1MB**。
- **构成**：由 **64 个连续的页** 组成（64 * 16KB = 1MB）。
- **为什么要有“区”？**
  - 如果一页一页地申请磁盘空间，数据在物理上会非常分散（随机 IO）。
  - InnoDB 为了让数据尽可能**顺序读写**，通常是一次性申请一个“区”（64个连续页）。即使 B+ 树做范围查询，也能利用磁盘预读特性。

**③ 段 (Segment) —— 逻辑管理单位**

- 段一般分为数据段、索引段、回滚段等：
  1. 索引段：存放B+树的非叶子节点的区的集合
  2. 数据段：存放B+树的叶子节点的区的集合
  3. 回滚段：存放的是回滚数据的区的集合
- **细节**：段是区的集合，但段不要求区必须是物理连续的。

------

### **小结第一部分**

到现在为止，我们已经拿到了一个 `student.ibd` 文件，并且把它切开，拿出了其中的 **一个页 (Page，16KB)**。

你的那一行行代码写入的 `User` 数据，就静静地躺在这个 16KB 的 **User Records** 区域里。

**接下来是最精彩的部分：**

在这 16KB 的内部，这一行数据到底是变成了一串什么样的二进制代码？这就是思维导图里的核心——**COMPACT 行格式**。

# **第二阶段：微观行记录结构 (COMPACT 行格式字节级拆解)**

#### 第一部分：包裹单 (记录的额外信息)

这部分是给 MySQL 自己看的，虽然不包含你的业务数据，但决定了数据能不能被正确读出来。

#### 1. 变长字段长度列表 (Variable Length Field List)

如果你的表里有 `VARCHAR`、`VARBINARY`、`TEXT` 这种长度不固定的类型，MySQL 必须记下来这一列到底有多长，否则读的时候不知道读到哪里结束。如果数据表里没有变长字段时，这时候表的行格式就不会有变长字段长度列表。

- **存什么**：存每个变长列的**实际字节长度**（十六进制）。

- **怎么存（核心考点）**：**逆序存放！**

  - **例子**：假设表定义是 `(col1 varchar, col2 varchar)`。
  - 第一行数据：`col1` 内容是 "a" (长度1)，`col2` 内容是 "bb" (长度2)。
  - 存的时候不是 `01 02`，而是 **`02 01`**。

  ![image-20260123202918022](C:\Users\lenovo\AppData\Roaming\Typora\typora-user-images\image-20260123202918022.png)

  ![image-20260123202939248](C:\Users\lenovo\AppData\Roaming\Typora\typora-user-images\image-20260123202939248.png)

- **为什么要逆序？**

  - 这是一个极为巧妙的设计。因为**记录头信息**（Record Header）里有一个指针指向下一条记录。
  - 把长度列表放在最前面且逆序，使得**记录头信息**恰好夹在“记录头信息”和“真实数据”中间。
  - CPU 在读取数据时，向左读就是长度，向右读就是数据，寻址效率最高（类似于 CPU 缓存行的局部性原理）。

  > ![image-20260123204311384](C:\Users\lenovo\AppData\Roaming\Typora\typora-user-images\image-20260123204311384.png)
  >
  > 因为 **“第 1 列的长度”** 和 **“第 1 列的数据”** 物理距离很近（都靠近中间的指针）。
  >
  > 所以，当 CPU 把指针附近的数据加载到缓存时，往往能**同时**把“长度”和“数据”都加载进来。
  >
  > 这大大提高了读取效率（局部性原理）。

#### 2. NULL 值列表 (NULL Bitmask)

如果你的表允许 `NULL`，MySQL 哪怕一个字节都不想浪费在 NULL 上。它用一个**二进制位图 (BitMap)** 来标记。

- **原理**：

  - 1 代表该列是 NULL。
  - 0 代表该列有值。

- **也是逆序的**。

- **例子**：

  - 假设你有 3 个允许为 NULL 的字段：`c1, c2, c3`。
  - 实际数据：`c1=NULL`, `c2=数据`, `c3=NULL`。
  - 二进制位图就是：`101`（注意顺序对应）。

  ![image-20260123205711189](C:\Users\lenovo\AppData\Roaming\Typora\typora-user-images\image-20260123205711189.png)

  - 转成十六进制存进去

  ![image-20260123205930885](C:\Users\lenovo\AppData\Roaming\Typora\typora-user-images\image-20260123205930885.png)

- **空间占用**：如果允许 NULL 的列少于 8 个，就只占 **1 字节**。如果超过 8 个，就占 2 字节，以此类推（按字节对齐）。

- **省空间大法**：

  - 在后面的“真实数据”区域，**所有值为 NULL 的列，是根本不占空间的！** 它们只在这个小小的列表里占了一个二进制位。
  - 在设计数据库表的时候，通常都是建议将字段设置为NOT NULL，这样可以至少节省1字节的空间

#### 3. 记录头信息 (Record Header) —— 5 字节

这是管理这行数据的元数据，里面有几个标志位非常重要：

- **`delete_mask` (1 bit)**：
  - 你执行 `DELETE` 语句时，数据真的被物理删除了吗？**没有！**
  - MySQL 只是把这个 bit 打上标记（置为 1）。
  - **为什么？** 为了性能。如果每次删除都重新排列磁盘上的数据，IO 就炸了。这些被标记的空间后面会被新插入的数据复用。
- **`next_record` (指针)**：
  - 记录这一行数据到**下一行数据**的地址偏移量。
  - 这也证明了：**页内的数据虽然物理上可能杂乱，但在逻辑上是一个单向链表。**
- **`record_type`**：
  - 0：普通数据记录。
  - 1：B+ 树非叶子节点的目录项。
  - 2：Infimum（最小记录）。
  - 3：Supremum（最大记录）。

#### 第二部分：包裹货品 (记录的真实数据)

这里不仅有你存的 `name`, `age`，还有 MySQL 强行塞进去的**私货（隐藏列）**。这三个隐藏列是面试中关于 **事务 (Transaction)** 和 **MVCC** 的核心考点。

#### 1. 隐藏列 (Hidden Columns)

不管你建表时怎么写，InnoDB 都会在你的数据前面加这三列：

![image-20260123212213846](C:\Users\lenovo\AppData\Roaming\Typora\typora-user-images\image-20260123212213846.png)

1. **DB_ROW_ID (6 字节)**：
   - **行 ID**。
   - 如果你没有设置主键，也没有唯一非空索引，InnoDB 就用它来当主键，构建聚簇索引 B+ 树。
   - *如果你设了主键，这一列就没有了。*
2. **DB_TRX_ID (6 字节) —— 必需！**
   - **事务 ID**。
   - 记录**是谁（哪个事务）**最后一次修改（Insert/Update）了这行数据。
   - 这是 **MVCC (多版本并发控制)** 判断数据可见性的核心依据。
3. **DB_ROLL_PTR (7 字节) —— 必需！**
   - **回滚指针**。
   - 指向 **Undo Log** 里这行数据的**上一个版本**。
   - 当你执行 `ROLLBACK`，或者在 MVCC 中需要查“旧版本数据”时，就是顺着这个指针找回去的。

#### 2. 用户真实数据

最后才是你定义的 `id`, `name`, `age`...

- **定长字段**（如 `char(10)`, `int`）：直接分配固定空间。
- **变长字段**（如 `varchar`）：按照实际长度存。
- **NULL 字段**：**这里不存！**（前面说了，在 NULL 值列表里标记了）。

### 小结第二阶段

如果我们有一张表：

```
CREATE TABLE t (
    id INT,
    name VARCHAR(10),
    job VARCHAR(10)
)ROW_FORMAT=COMPACT;
```

插入数据：`(1, "a", NULL)`

那么这行数据在磁盘上的样子大概是（十六进制示意）：

| **区域**     | **内容**            | **含义**                                                   |
| ------------ | ------------------- | ---------------------------------------------------------- |
| **变长列表** | `01`                | 只有 `name` 是变长且非NULL，长度为1。(`job`是NULL不记长度) |
| **NULL列表** | `02` (二进制`10`)   | 第二个允许NULL的列(`job`)是NULL，第一个(`name`)不是。      |
| **记录头**   | `00 00 ...`         | delete_mask=0, next_record指向下一行...                    |
| **隐藏列**   | `TrxID` + `RollPtr` | 事务ID和回滚指针                                           |
| **真实数据** | `00 00 00 01`       | id = 1 (INT占4字节)                                        |
| **真实数据** | `61`                | name = "a" (ASCII码 'a' 是 0x61)                           |
| **真实数据** | (无)                | job 是 NULL，这里完全不占空间                              |

这就是 **COMPACT 行格式** 的全部秘密。

# 第三阶段：边界与极端场景

#### 第一部分：Varchar(n) 的n最大值为多少

很多面试题问：“Varchar 最大能存多少个字符？”

如果你直接回答 “65535”，那就掉进陷阱了。

**MySQL 有一条铁律：**

> **MySQL 表中的一行记录（Row），其所有列的数据总长度，不能超过 65535 字节。**
>
> （注意：是字节，不是字符；是整行，不仅仅是单列）

你需要像做算术题一样把这个空间算出来。

**公式：**

```
65535 字节 - [变长字段长度列表] - [NULL 值列表] - [记录头信息] = [真实数据可用空间]
```

##### **场景 1：单字段的情况 (只有一列 Varchar)**

假设我们创建一张表，只有一个 `name` 字段，允许为 NULL，字符集是 `utf8mb4`（常用，1 个字符最多占 4 字节）。

```
CREATE TABLE t (name VARCHAR(N)) CHARSET=utf8mb4;
```

我们来算 N 最大能填多少：

1. **减去 NULL 列表**：允许 NULL，占用 **1 字节**。

2. **减去 长度列表**：因为 N 肯定很大（超过 255），需要 **2 字节** 来记录长度。

3. **剩多少给数据？**

   `65535 - 1 - 2 = 65532 字节`。

4. **换算成字符数 (n)**：

   `65532 / 4 = 16383`。

**结论**：在 `utf8mb4` 下，N 最大只能是 **16383**。如果你写 `varchar(20000)`，建表直接报错。

> *注：如果是 `latin1` (1字节/字符)，那 N 就是 65532。*

##### **场景 2：多字段的情况 (共享 65535)**

如果有多个字段的话，要保证所有字段的长度+变长字段字节数列表所占用的字节数+NULL值列表所占用的字节数<=65535

**核心概念**：这个 65535 是**全家共享**的额度。

------

#### 第二部分：行溢出 (Row Overflow) —— 页不够装了怎么办？

这里有一个巨大的矛盾点，请仔细看：

- **矛盾**：
  - MySQL 规定：一行记录最大可以有 **65535 字节**（我们刚才算的）。
  - MySQL 规定：一个页（Page）默认只有 **16384 字节 (16KB)**。

**问题来了：** 一页纸（16KB）根本写不下一行字（65535字节），这不物理违规了吗？

这就是 **“行溢出”**。

当一行数据太大，当前页（主页）放不下时，MySQL 就会把数据切分，存到其他的页（溢出页 / Blob Page）里去。

这里根据行格式的不同（旧 vs 新），有两种处理方式：

##### **1. COMPACT 格式 (旧时代的妥协)**

这是 MySQL 5.1 时代的逻辑。它不想把鸡蛋全放在别的篮子里。

- **处理方式**：

  - 在当前页（主页）里，**硬挤出 768 字节** 来存数据的前面一小部分。
  - 剩下的数据存到溢出页。
  - 当前页末尾留一个 **20 字节的指针** 指向溢出页，大致如下图所示：

  ![image-20260123215955339](C:\Users\lenovo\AppData\Roaming\Typora\typora-user-images\image-20260123215955339.png)

- **缺点**：因为你在主页里存了 768 字节的“肉”，导致主页能存放的“目录（行记录）”变少了。B+ 树变高了，查询变慢。

##### **2. DYNAMIC 格式 (新时代的默认)**

这是 MySQL 5.7 和 8.0 的默认格式，非常干脆。

- **处理方式**：

  - 当前页（主页）里，**连 1 个字节的真实数据都不存**。
  - 只存那个 **20 字节的指针**。
  - 所有的数据（100%）都扔到溢出页里去，大致如下图所示：

  ![image-20260123220031925](C:\Users\lenovo\AppData\Roaming\Typora\typora-user-images\image-20260123220031925.png)

- **优点**：

  - 主页变得非常“瘦”。哪怕你这一行有 60000 字节，在主索引页里也只占 20 字节。
  - 这意味着一页能存成千上万个指针，B+ 树会变得很扁平，索引效率极高。

------

### 终极总结

1. **宏观**：数据存在 `.ibd` 文件的 **页 (Page)** 里。原子 DDL 靠系统表空间保证。
2. **微观**：一行数据由 **[变长长度(逆序) + NULL表(逆序) + 头信息 + 隐藏列 + 真实数据]** 组成。
3. **边界**：
   - **限制**：整行不能超 65535 字节（utf8mb4 下 varchar 顶多 16383）。
   - **溢出**：如果你真的存满了，**DYNAMIC** 格式会只留一个 20 字节的地址条在家里，把货物全搬到仓库（溢出页）去，以此保证索引速度。

这套知识体系（从文件到二进制位），已经是 MySQL 存储引擎的最底层逻辑了。如果面试能答到这个深度，基本是满分。



# Q&A

> ## **MySQL 的 NULL 值是怎么存放的？**

MySQL 的 Compact 行格式中会用「NULL值列表」来标记值为 NULL 的列，NULL 值并不会存储在行格式中的真实数据部分。

NULL值列表会占用 1 字节空间，当表中所有字段都定义成 NOT NULL，行格式中就不会有 NULL值列表，这样可节省 1 字节的空间。



> ## **MySQL 怎么知道 varchar(n) 实际占用数据的大小？**

MySQL 的 Compact 行格式中会用「变长字段长度列表」存储变长字段实际占用的数据大小。



> ## **varchar(n) 中 n 最大取值为多少？**

<u>如果一张表只有一个 varchar(n)  字段</u>，且允许为 NULL，字符集为 ascii。varchar(n) 中 n 最大取值为 65532。计算公式：65535 - 变长字段字节数列表所占用的字节数 - NULL值列表所占用的字节数 = 65535 - 2 - 1 = 65532。

<u>如果有多个字段的话</u>，要保证所有字段的长度 + 变长字段字节数列表所占用的字节数 + NULL值列表所占用的字节数 <= 65535。



> ## **行溢出后，MySQL 是怎么处理的？**

如果一个数据页存不了一条记录，InnoDB 存储引擎会自动将溢出的数据存放到「溢出页」中。

<u>Compact 行格式</u>针对行溢出的处理是这样的：当发生行溢出时，在记录的真实数据处只会保存该列的一部分数据，而把剩余的数据放在「溢出页」中，然后真实数据处用 20 字节存储指向溢出页的地址，从而可以找到剩余数据所在的页。

<u>Compressed 和 Dynamic 这两种格式</u>采用完全的行溢出方式，记录的真实数据处不会存储该列的一部分数据，只存储 20 个字节的指针来指向溢出页。而实际的数据都存储在溢出页中。
